# Consilium Plan - Agent Docs Easy Install

**Slug:** agent-docs-easy-install
**Date:** 2026-06-20
**Status:** consilium-complete
**Feature:** `документация для агентов и возможность легкой установки для codex/claude/openclaw`
**Basis:** INVARIANTS §1 harness=files, §2 task-based skills, D-002 documented OAuth, D-008 Mail complete, current `.claude` + `.codex` install surface.

Type-2 plan. Owner sign-off required before `/implementor agent-docs-easy-install`.

## TL;DR

- **Severity:** HIGH 8, MEDIUM 12, LOW 4.
- **Top must-fix 1:** Confirm whether `openclaw` means **OpenCode** or a separate **OpenClaw** runtime. Current repo contract says Claude Code / OpenCode / Codex; OpenClaw support is unverified.
- **Top must-fix 2:** Fix stale agent entrypoints before adding install docs. `CLAUDE.md` still says token interception/private endpoints, contradicting D-002/D-004/D-008.
- **Top must-fix 3:** Design a runtime-neutral agent pack with per-client adapters, not a workflow wrapper. Installer may provision/doctor/sync files only; it must not run feature work.

## Blockers

- **B1 - Runtime target ambiguity (`openclaw` vs `opencode`).** The request names `openclaw`, while project rules and invariants name OpenCode/Codex/Claude Code. Decide if this feature targets OpenCode, a real OpenClaw CLI, or both. If OpenClaw is real, support must start as experimental until a canonical docs URL and smoke test exist.
- **B2 - Stale `CLAUDE.md` would poison installed docs.** [CLAUDE.md](/Users/sbeysenov/dev/yx360-cli/CLAUDE.md:3) still describes intercepted web/mobile session tokens and private endpoints. Rewrite it around documented OAuth, Mail IMAP/SMTP, and documented-first escalation before making it an install artifact.
- **B3 - Setup/install scope must stay provisioning-only.** INVARIANTS §1.1 forbids a dev-workflow wrapper. Any new installer/sync/doctor commands may copy, merge, render, validate, and lock files; they must not invoke `/pre-feature`, `/implementor`, “fix”, “build feature”, or agent workflows.
- **B4 - Current setup path is Claude-only.** `.claude/settings.json` and `.claude/skills/setup/SKILL.md` provision Claude artifacts, while `.codex/` is untracked and OpenCode/OpenClaw has no adapter. A cross-runtime install claim needs an explicit client matrix plus per-client smoke tests.
- **B5 - Session mining privacy needs opt-in.** `.claude/skills/setup/SKILL.md` currently mines prior Claude/OpenCode/Codex session files before the install interview. Cross-runtime install must make session-history mining explicit opt-in with redaction and ignored raw reports.

## Concerns

- **C1 - Absolute Codex hook path is non-portable.** `.codex/hooks.json` embeds `/Users/sbeysenov/dev/yx360-cli/...`; target projects would point back at this checkout. Render target-local paths during install or use a verified relative command.
- **C2 - `.codex/` is untracked.** Codex support currently exists in the working tree but not the repository; it will not ship unless tracked or generated by setup.
- **C3 - `.harness-lock` does not cover Codex/OpenCode artifacts.** Lock metadata should include every generated surface artifact, including `.codex`, `.opencode`, `.agents`, and future adapter files, with owner/template/rendered checksums.
- **C4 - README/onboarding mismatch.** AGENTS.md references `README.md` as onboarding/manual install guide, but no README exists. Either create it or correct the repository map.
- **C5 - Repository map has stale file counts and missing tech docs.** AGENTS.md claims `.claude/agents/` has 12 files and tech-details has 9 files; the current tree differs. Cross-runtime docs should avoid fixed counts or regenerate the map.
- **C6 - Monolithic instructions risk truncation/adherence drift.** Codex and Claude have project-memory size/adherence limits; keep root `AGENTS.md` concise and move workflows into task-based skills/adapters.
- **C7 - Safety controls cannot live only in Markdown.** Where CLIs support native permission/config controls, installers should provision deny/default permission settings there too, while documenting Markdown as guidance.
- **C8 - Skill directory discovery differs by runtime.** Author portable `SKILL.md` folders once, then install/copy/symlink into runtime-specific locations (`.claude/skills`, `.agents/skills`, `.opencode/skills`, Codex plugin/skill path) based on the chosen matrix.
- **C9 - Sync propagation is tricky.** Existing setup/sync avoids overwriting project templates; runtime adapter migrations need explicit migration notes and conflict handling, not silent overwrites.
- **C10 - Destructive adapter reshuffles need migration manifests.** Runtime adapter deletes/renames should be treated as conflicts or migration steps, not ordinary auto-deletes.
- **C11 - JSON output contract is undocumented.** Agent docs should not scrape human output. Add a small stable agent command contract for `--json` payloads and token-safety rules.
- **C12 - Plaintext token store gets riskier with agent use.** If docs encourage agents to use Mail scopes, warn strongly about `--insecure-file-store`; consider refusing send-capable scopes with plaintext storage unless explicitly allowed in a later decision.

## Proposed Architecture

### Source Of Truth

- **Canonical instructions:** `AGENTS.md` remains the main project/runtime-agnostic working agreement.
- **Claude shim:** `CLAUDE.md` becomes short and current; preferably imports or points to `AGENTS.md` plus Claude-only deltas.
- **Agent command contract:** add `docs/agent-contract.md` or `AGENTS.md` section documenting stable CLI usage for agents: auth prerequisites, `--json`, redaction, error semantics, and safe send behavior.
- **Runtime matrix:** add docs mapping canonical artifacts to each supported client:
  - Codex: `AGENTS.md`, `.codex/hooks.json`, `.codex/hooks/inject-state.sh`, optional Codex skill/plugin packaging.
  - Claude Code: `CLAUDE.md`, `.claude/settings.json`, `.claude/agents`, `.claude/skills`, hooks.
  - OpenCode: `AGENTS.md`, `opencode.json` / `.opencode`, `.agents/skills` where supported.
  - OpenClaw: experimental until canonical docs and smoke test exist.

### Installer Shape

- Add provisioning-only commands/scripts or skill steps:
  - `install` / `/setup`: render selected runtime adapters, never run dev workflows.
  - `doctor`: verify each selected runtime can see expected files/hooks/skills.
  - `sync`: update owned artifacts with lock/drift/conflict reporting.
- Installer modes:
  - `--project`: write committed project files.
  - `--user`: write user/global files only after explicit confirmation.
  - `--dry-run`: default for non-empty targets; show file plan and conflicts.
- Locking:
  - Extend `.harness-lock` for all generated adapters with owner, template source, rendered path, sha256, and whether content is target-rendered.

### Verification Gate

- Static:
  - `go test ./...`, `go vet ./...`, `go build -o bin/yx360 ./cmd/yx360` if installer code changes Go.
  - Validate JSON/YAML/TOML config syntax.
  - Run shellcheck or `bash -n` for hook scripts.
- Smoke:
  - Claude: verify `.claude/hooks/inject-state.sh` runs from a copied target.
  - Codex: verify rendered `.codex/hooks.json` points to the target-local hook and the hook runs.
  - OpenCode/OpenClaw: verify documented config exists and can be discovered; if not executable locally, mark unsupported/experimental.
  - Agent contract: run at least one `yx360 --json` or `mail list --json` fixture test without leaking tokens.

## Proposed PR Slicing

- **PR-agent-docs-1:** Fix stale docs: `CLAUDE.md`, AGENTS repository map, missing README/onboarding, agent command contract, runtime compatibility matrix.
- **PR-agent-install-2:** Make adapter templates/rendering portable: `.codex` target-local hook path, lock coverage, setup copy matrix, per-runtime dry-run.
- **PR-agent-doctor-3:** Add `doctor`/verification: per-runtime smoke checks, hook syntax checks, unsupported runtime reporting.
- **PR-agent-skills-4:** Normalize portable skills: source-of-truth skill folders and per-runtime install locations/symlink/copy rules.

## Out-of-Scope (Declared)

- No command that asks harness/yx360 to perform development work (`harness fix`, `harness implement`, `yx360 agent run feature`).
- No global config writes without explicit approval.
- No automatic reading of prior agent session histories without opt-in and redaction.
- No hard claim of OpenClaw support until canonical OpenClaw docs and a real smoke test exist.
- No rewrite of the whole harness file layout in the same PR as docs/install fixes.

## Open Questions Raised

- **OQ-011 - OpenClaw target identity.** Does “openclaw” mean OpenCode, a separate OpenClaw CLI, or both?
- **OQ-012 - Agent install distribution mode.** Should easy install be a Go subcommand, a shell script, a harness `/setup` enhancement, a Codex/Claude skill, or a combination? All allowed variants must remain provisioning-only.
- **OQ-013 - User/global install policy.** Which files may be written outside the target repo, and what confirmation/dry-run rules are required?

- **Runtime-specific enforced permissions/configs differ from Markdown guidance.** Claude/OpenCode have settings/permission surfaces; do not rely only on instruction files for safety. Source date 2026-06-20; confidence corroborated.
- **Skill discovery differs across tools.** Claude uses `.claude/skills`; OpenCode can search `.opencode/skills`, `.claude/skills`, and `.agents/skills`; Codex uses skills/plugins in its own model. Source date 2026-06-20; confidence corroborated for Claude/OpenCode, medium for Codex local plugin path specifics.
- **OpenClaw docs were not found in authoritative form during this pass.** Treat as unverified/experimental. Source date 2026-06-20; confidence unverified.
